# Project Agent Instructions

This file is the project-local contract for AI-assisted development after
SpecOps Overlay is adopted by a repository. Keep it short, concrete, and
project-specific. Put reusable workflow guidance in `docs/`, repository facts
in `docs/project/`, reusable workflows in `skills/`, and stack-specific
guidance in the selected flavor when one applies.

This repository is the SpecOps Overlay core for OpenSpec. It is not a
runnable service by itself. The fields below are defaults and placeholders until
the adopting project fills them with real commands, paths, dependencies, and
constraints.

Tool-specific OpenSpec command and skill files are generated by `openspec init`
or `openspec update` in each adopting repository. Do not maintain generated
`.claude/` or `.github/prompts/opsx-*` files as overlay source.

## Overlay Identity

- The repository root is the SpecOps Overlay core. Do not add a nested
  `core/` directory.
- OpenSpec owns proposal, spec, task, apply, sync, and archive structure.
  SpecOps Overlay adds project context, quality gates, AC-to-proof
  traceability, scope discipline, and post-implementation evaluation.
- Stack guidance is optional and belongs in flavors under `flavors/<id>/` when
  present. Java/Quarkus is the first supported flavor, not a requirement for
  adopting the overlay core.
- Generic adoption must work without assuming any language, framework, build
  tool, database, container runtime, or cloud provider.

## How To Adopt SpecOps Overlay

1. Copy the overlay files into the root of the adopting repository.
2. Fill `Template Defaults / Fill After Adoption` above with real project
   facts.
3. Fill every file under `docs/project/` with the adopting project's
   architecture, stack, structure, conventions, testing, integrations, and
   concerns.
4. For existing repositories, use `skills/brownfield-mapping/SKILL.md` to fill
   `docs/project/*` from observed code, manifests, scripts, and CI before
   relying on agents for implementation.
5. Confirm `openspec/config.yaml` matches the repository's desired OpenSpec
   workflow and context rules.
6. Run `openspec init` in the adopting repository and select the AI tools that
   should receive generated OpenSpec commands and skills.
7. Run `openspec config profile` and `openspec update` if the project needs the
   expanded OPSX workflow commands beyond the default core profile.
8. Load stack flavor guidance only when the adopting repository uses that stack.
   When a flavor is selected, inject its `AGENTS.patch.md` fragment between the
   `<!-- FLAVOR:BEGIN -->` and `<!-- FLAVOR:END -->` markers below.
9. Keep reusable workflow guidance in `docs/`, repository facts in
   `docs/project/`, reusable workflows in `skills/`, and stack-specific
   guidance in `flavors/<id>/` when present.

## Local Runtime

- Prefer the project wrapper or documented build command over globally installed
  tools.
- Start required infrastructure before running the app or integration tests.
- Do not assume Docker, containers, databases, queues, or credentials are
  available until the repository documents or local scripts confirm them.
- Use project-local configuration files and profiles. Do not invent environment
  variables unless the project already follows that pattern.

## Architecture Principles

The default architecture is modular and domain-oriented:

- Applications/services are bootstraps and orchestration only.
- Business logic lives in explicit domain modules.
- A module represents a bounded context with its own vocabulary, model,
  persistence ownership, and public API.
- One aggregate or business concept maps to one cohesive unit of code.
- Cross-module communication goes through explicit public APIs, facades,
  ports, clients, events, or commands. Do not reach into another module's
  internals.
- Keep API/controller/handler classes lean. Domain decisions belong in services.
- Keep persistence details inside repositories/adapters. Services should call
  business-meaningful methods instead of persistence-specific APIs.
- Shared code must be small and stable. Do not create a shared kernel for
  business logic just because two modules need similar behavior.

## Skill Gates

Use skills selectively as quality gates. Do not load every skill by default.

| Gate | When to Use | Outcome |
| --- | --- | --- |
| `skills/brownfield-mapping/SKILL.md` | When adopting SpecOps Overlay in an existing repository or filling `docs/project/*` from real code | Evidence-based project reference docs and explicit unknowns |
| Selected flavor architecture skill, e.g. `flavors/java-quarkus/skills/modular-architecture/SKILL.md` | Before designing or implementing modules, boundaries, facades, repositories, persistence ownership, or subdomains when the selected flavor provides one | Architecture design/checklist and boundary verification |
| `skills/spec-quality-gate/SKILL.md` | Before implementation when a change has explicit ACs, hidden requirement risk, integration, persistence, messaging, security, config, async, or webhook risk | Pass/partial/fail planning gate with AC proof matrix, test strategy, and scope decisions |
| `skills/spec-driven-eval/SKILL.md` | After implementation when there is a PRD, OpenSpec change, acceptance criteria, or explicit spec to score against | Evidence-based implementation and test coverage score |
| `skills/pr-review/SKILL.md` | Only when explicitly reviewing a PR, branch, or diff | Consolidated PR findings ordered by risk |

For OpenSpec work, prefer:

```text
adopt/map      -> brownfield-mapping -> docs/project/*
proposal/specs -> docs/project/* + openspec/config.yaml
design/tasks   -> add spec-quality-gate when ACs or hidden-risk proof is needed; add selected flavor architecture skill only when structure is affected
apply          -> AGENTS.md + relevant docs/project/* + stack-specific patterns from flavor
verify/archive -> add spec-driven-eval when acceptance criteria are clear
pre-merge PR   -> pr-review
```

## OpenSpec Task Discipline

- Before naming delta specs, inspect `openspec/specs/` for existing capability
  names. Use MODIFIED deltas for changed existing behavior and ADDED deltas for
  new behavior. Each capability maps to
  `openspec/changes/<change-name>/specs/<capability>/spec.md`.
- Run `skills/spec-quality-gate/SKILL.md` by default before implementation
  when a change has explicit ACs or touches persistence, messaging, external
  integrations, security, config, async, scheduled, or webhook behavior. If the
  gate is skipped, record the opt-out rationale.
- Implementation tasks must include task-local `Tests`, `Gate`, `Done when`,
  and `Maps to` fields. Map each task to an AC, sanctioned implicit risk, or
  engineering gate.
- For code changes, write or update the smallest appropriate failing test
  first, confirm the failure when feasible, then implement the smallest passing
  change. If RED cannot be run, record the reason.
- Do not delete tests, skip tests, weaken assertions, lower test scope, or
  mark failures acceptable merely to pass validation. Any exception requires
  explicit rationale and AC/risk/gate mapping.
- Use per-task validation gates: `quick`, `full`, `build`, `docs`,
  `security`, `OpenSpec verify`, or `evaluation`.
- Prefer one focused Conventional Commit per non-trivial implementation task
  when the repository workflow allows it, such as `feat(api): add invite
  endpoint` or `docs(tasks): add proof mapping`.
- Before archive or merge, run `/opsx:verify` when the expanded profile is
  available, then run `skills/spec-driven-eval/SKILL.md` when acceptance
  criteria are clear enough to score. For the core profile, record
  `/opsx:verify` as unavailable and run `openspec validate <change-id>` or the
  closest generated OpenSpec structural guidance.
- Do not archive until specs are synced or explicitly archive-ready, evaluation
  evidence is recorded, and unresolved gaps are named.

## Change Sizing

Use the lightest OpenSpec path that still preserves traceability and risk
control.

| Size | Use For | Artifacts | Gates |
| --- | --- | --- | --- |
| Small | Typo, local docs tweak, tiny refactor with no behavior change | Short change/task note when traceability matters | `quick` or `docs` |
| Medium | Single behavior or template change | Proposal, delta spec when behavior changes, focused tasks | Quality gate when ACs or risks apply |
| Large | Multi-file behavior, integration, persistence, security, or workflow change | Proposal, specs, design, tasks, AC proof matrix | Quality gate default-on, verify, eval |
| Complex | Cross-module, async, migration, rollout, or high uncertainty | Full artifact set with frozen assumptions | Quality gate required, full validation |

Small changes may use lighter artifacts, but they must not bypass
`openspec/changes/`, AC/risk/gate mapping, validation evidence, or sync/archive
expectations. Risky surfaces still trigger the quality gate regardless of size.
Record "while here" ideas as out of scope or deferred unless they map to an AC,
sanctioned implicit risk, or engineering gate.

## Knowledge Verification

For research-heavy design or brownfield work, verify facts in this order:

1. Local code, manifests, scripts, tests, and CI.
2. Filled `docs/project/*` reference docs.
3. Selected flavor docs and skills, only when the stack matches.
4. Official vendor or framework documentation.
5. Web search only when current external facts are needed.

Record uncertainty explicitly. Do not invent commands, dependencies, APIs,
environment variables, or architecture decisions.

## State And Handoff

For long-running changes, record compact resume context in the OpenSpec change
itself, preferably in `design.md` or `tasks.md`: decisions, blockers,
validation results, deferred ideas, and remaining archive steps. Do not create
a parallel workflow outside OpenSpec.

## Implementation Rules

- Read existing code before changing it. Follow local patterns unless they
  conflict with these instructions or the task explicitly asks for a change.
- Keep edits scoped to the requested behavior.
- Prefer established conventions of the project stack over clever abstractions.
- Add e2e or integration coverage for important API, persistence, messaging, or
  cross-module behavior.
- Add unit coverage for business rules and edge cases.
- Do not create database migrations manually when the project has a migration
  generation workflow. Use the repository's documented migration process.
- Do not store secrets in source files, test fixtures, logs, or docs.

## Plans and Verification

Implementation plans should include the relevant checks for the project:

- Build: use the documented build command from `docs/project/TESTING.md`.
- Static checks: use the documented linters and analysis tools.
- Unit tests: use the documented unit test command.
- Integration/e2e tests: use the documented integration/e2e test command.
- Migration validation when persistence changes.

If a command is unknown, inspect `README.md`, build manifests, CI
configuration, or project scripts before guessing.

## OpenSpec Integration

- Treat `openspec/config.yaml` as the OpenSpec context bridge. It should point
  to the project reference docs and define rules for OpenSpec artifacts.
- Do not copy this entire `AGENTS.md` into `openspec/config.yaml`; that creates
  drift. Keep `AGENTS.md` as the agent contract and use `openspec/config.yaml`
  for OpenSpec artifact context.
- Put current observable behavior in `openspec/specs/`.
- Put proposed changes in `openspec/changes/<change-name>/`.
- Keep architecture, stack, structure, testing, integrations, conventions, and
  concerns in `docs/project/` unless they define behavior that should be
  specified.
- Use `templates/` when creating proposal, spec, design, task, AC proof, or
  evaluation artifacts that do not already have a more specific project
  template.
- Use the selected flavor architecture skill, e.g.
  `flavors/java-quarkus/skills/modular-architecture/SKILL.md`, as a conditional
  design and verification gate when a change affects module boundaries or
  persistence ownership and the selected flavor provides one.
- Use `skills/spec-quality-gate/SKILL.md` before implementation when a change
  has explicit ACs, integration/persistence/security/async risk, or hidden
  requirement risk that needs AC-to-proof traceability.
- Run `/opsx:verify` before `skills/spec-driven-eval/SKILL.md` when the
  expanded OpenSpec profile is available. If it is unavailable, record that
  status and use `openspec validate <change-id>` or equivalent generated
  OpenSpec structural review before evaluation.
- Use `skills/spec-driven-eval/SKILL.md` as a conditional verification gate
  before archive/merge when acceptance criteria are clear enough to score.
- Keep `skills/pr-review/SKILL.md` out of normal OpenSpec planning; use it for
  explicit PR or diff review.
